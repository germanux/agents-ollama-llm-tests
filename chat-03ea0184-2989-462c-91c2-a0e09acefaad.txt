PERSISTENT SPRING BOOT STARTUP

Do not use any of these patterns:

- `timeout ... java -jar ...`
- `java -jar ... &`
- `nohup java -jar ... &`
- `pkill java`
- `killall java`

Those approaches either terminate the server when the tool call ends, kill it after a timeout, or risk killing unrelated Java processes.

Run Spring Boot as a persistent user-level transient systemd service.

First stop only the previous benchmark service, if it exists:

systemctl --user stop benchmark-agent.service 2>/dev/null || true
systemctl --user reset-failed benchmark-agent.service 2>/dev/null || true

Then start the packaged JAR:

systemd-run --user \
  --unit=benchmark-agent \
  --collect \
  --working-directory="$ROOT" \
  "$JAVA_HOME/bin/java" \
  -jar "$JAR"

Explanation:

- `--user` creates the service under the current user's systemd instance. It does not require sudo.
- `--unit=benchmark-agent` creates `benchmark-agent.service`.
- `--collect` allows systemd to discard the transient unit after it has stopped.
- `--working-directory="$ROOT"` ensures Spring starts from the project directory.
- `"$JAVA_HOME/bin/java"` guarantees Java 21 is used instead of another Java version.
- The process is owned by systemd, so it remains alive after the Bash tool call finishes.
- Never start a second copy while `benchmark-agent.service` is already active.
